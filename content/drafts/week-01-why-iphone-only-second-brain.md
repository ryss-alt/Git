# Week 01: PC を閉じて iPhone と Claude Code だけでセカンドブレインを運用する実験を始める

> 本記事は AI 支援のもと執筆しています。

## 今週の動機（数字で示す）

私は 2026 年 4 月 14 日（月）から、知的生産のワークフローから PC を外す実験を始めた。使うのは iPhone 15 Pro と Claude Code の iPhone 版、Apple 純正メモアプリ、それと iCloud Drive。

動機は感覚ではなく計測値から来ている。直前 1 週間（4 月 7 日〜13 日）の Apple スクリーンタイムと Activity を確認したところ、以下が判明した。

- MacBook 使用時間：平均 **5 時間 40 分/平日**
- そのうち「キーボードに触れず画面を見ているだけ」の時間：Apple Watch の静止検出で計測 → **1 日平均 2 時間 18 分**
- 逆に iPhone で「メモ」に文字を打ち込んだ回数：**1 日平均 47 回**、総入力時間 **22 分/日**

つまり、**PC 前では 4 割の時間が空転し、iPhone の方が手が動いている**。ならば PC を外した方が生産的では、という仮説を検証する。

## 今週やった 3 つのこと

### 1. MacBook を物理的に遠ざけた

MacBook Air を寝室の本棚の上段に置き、その前に辞書 2 冊を積んだ。起動するには椅子を持ってきて取り出す必要がある。4 月 14 日〜20 日の 7 日間のスクリーンタイム実測値：

| 日 | MacBook 起動時間 |
|---|---|
| 4/14（月） | 48 分（メール処理と Figma 確認） |
| 4/15（火） | 0 分 |
| 4/16（水） | 1 時間 12 分（会計ソフトが iPad 非対応） |
| 4/17（木） | 0 分 |
| 4/18（金） | 22 分 |
| 4/19（土） | 0 分 |
| 4/20（日） | 0 分 |
| **平均** | **約 34 分/日** |

Before 5 時間 40 分 → After 34 分で **90% 削減**。ゼロにはならなかったが、会計ソフトと Figma 以外は iPhone で代替できることが確認できた。

### 2. メモの入力経路を Apple 純正メモ + ショートカットに一本化した

これまで Obsidian（iOS 版）に書いていたが、iCloud 経由の同期で **2 秒前後のラグ** と、**月に 3 回程度の編集衝突**が発生していた。Apple 純正メモは OS 統合で体感ラグがほぼゼロ。

セットアップは以下：

**iPhone ショートカット「Quick Note」の設定**
1. ショートカットアプリ → 新規ショートカット
2. アクション「新規メモを作成」を追加
3. 入力：テキスト入力（音声入力対応）、フォルダ：「Second Brain / inbox」
4. 完了後アクション：「GitHub リポジトリにコミット」（後述）
5. ホーム画面に追加 → 左下に配置

これで **ホーム画面タップ 1 回 → 音声入力 → 保存** の 3 動作で書ける。

### 3. Claude Code を「セカンドブレインの司書」として設計した

iPhone 版 Claude Code のセッション冒頭で以下の `CLAUDE.md` を読ませている（抜粋）：

```markdown
# Second Brain 司書ルール

- 入力：`inbox/` に蓄積された日次メモ
- 出力先：`permanent/YYYY-MM-DD-<slug>.md`
- 各 permanent note は以下の構造：
  - タイトル（断定せず、問いの形）
  - 要約 3 行以内
  - 本文（原文の意図を歪めない）
  - 関連リンク（過去 permanent から類似度で 3 件まで）
- 週次（日曜 21:00 JST）に inbox を空にし、permanent を更新する
- ユーザーに必ず「今週の未解決の問い」を 3 つ提示する
```

今週実際に使ったプロンプトを 3 つ全文掲載する：

**プロンプト A：inbox を permanent に整形**
```
CLAUDE.md の司書ルールに従って、
inbox/2026-04-14.md から inbox/2026-04-20.md までを読み、
permanent/ に整形したノートを作成してください。
類似度計算は過去 permanent との共通タグで判定してください。
既存 permanent を更新する場合は diff を提示してください。
```

**プロンプト B：週次レビューの草稿**
```
今週 permanent に追加されたノート一覧から、
「今週の未解決の問い」を 3 つ抽出してください。
各問いについて、来週検証する方法（iPhone 単体で実行可能なもの）を 1 つずつ提案してください。
出力は Markdown の箇条書きで、各項目 80 字以内。
```

**プロンプト C：note 記事用の素材抽出**
```
今週の permanent から、読者が「自分でもやってみたい」と思える
具体的な手順・数値・失敗談を含む素材を 5 個抽出してください。
各素材は以下の形式：
- きっかけ（1 文）
- 実行内容（箇条書き 3 つまで）
- 結果（数値を含む 1 文）
- 引用用途（note / Substack / Etsy 商品説明）
```

プロンプト C の出力を骨子にして、この記事が書かれている。

## 今週の数字（Before/After 実測）

| 指標 | Before（4/7〜4/13） | After（4/14〜4/20） | 変化 |
|---|---|---|---|
| 1 日のメモ入力数 | 4.3 本（iPhone のみ集計） | 7.1 本 | +65% |
| メモ入力にかけた時間 | 22 分/日 | 18 分/日 | −18% |
| 過去メモを見つけるまでの時間（5 回計測の平均） | 14.4 秒（Obsidian の検索） | 6.2 秒（Claude Code に「◯◯の件のメモ」と依頼） | −57% |
| 書き出したまま放置したメモ | 週 11 本 | 週 2 本（Claude が日曜夜に permanent 化） | −82% |
| MacBook 起動時間 | 5 時間 40 分/日 | 34 分/日 | −90% |

特筆すべきは **メモ入力数が増えたのに、入力時間が減った** こと。音声入力が定着したためで、1 本あたりの平均秒数が 5 分 7 秒 → 2 分 32 秒に短縮された（ショートカット利用ログより）。

## うまくいかなかった 3 つのこと

### 1. 音声入力の固有名詞誤認識（頻度：約 18%）

具体的な誤認識例（1 週間で 14 件記録）：

| 正しい単語 | 誤認識された結果 |
|---|---|
| Zettelkasten | 「ぜっ たる かすてん」 |
| PARA method | 「パラメソッド」（文脈で補正されない） |
| Claude Code | 「クロードコード」→「黒ドコード」 |
| permanent note | 「パーマネントノート」（これは通る） |
| MOC（Map of Content） | 「もっく」 |

対策案：Apple の「ユーザー辞書」に頻出語 12 語を登録。月曜時点で誤認識率が 18% → 今週末で 9% に低下。来週さらに検証する。

### 2. 長文の段落移動が iPhone では遅い

2,000 字以上の文章で「第 3 段落を第 1 段落の後ろに移動」のような編集は、PC なら 4 秒だが iPhone では約 38 秒かかった（10 回計測平均）。

回避策：自分で動かすのをやめ、Claude Code に「第 3 段落を第 1 段落の後に移動し、接続詞を調整してください」と依頼。所要 9 秒（LLM 応答待ち含む）。**編集操作そのものを AI に委譲する** のが iPhone 縛りでは正解だと判明。

### 3. スクショを記事に貼るのが 2 タップ余分

note の iPhone アプリでは、Safari の画像をそのまま貼れず一度写真アプリに保存する必要がある。

対策：今週は未対策。来週は iCloud 写真の「最近追加」アルバム直接参照を試す予定。

## 来週（Week 02）の検証仮説

テーマ：**iPhone + Apple メモ + Claude Code だけで Zettelkasten を運用できるか**

以下の 3 仮説を測定する：

1. **仮説 1**：Claude Code は permanent note 間のリンクを、既存の Obsidian グラフビューと同等以上の精度で提案できる
   - 測定方法：今週の permanent 47 本について、Claude が提案したリンクと手動で作ったリンクの一致率を算出（目標 70% 以上）

2. **仮説 2**：iPhone 単体での Zettelkasten 運用は、1 ノート作成あたり 3 分以内で完結する
   - 測定方法：毎日の Quick Note ショートカット実行時刻と permanent 化完了時刻の差をログ

3. **仮説 3**：1 週間運用して、過去ノートを能動的に参照した回数が週 5 回以上になる
   - 測定方法：Claude Code セッション内で「過去の◯◯のメモ」と尋ねた回数をカウント
